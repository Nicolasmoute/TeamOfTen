# t-2026-05-15-581e8c96

## Problem

`coord_commit_push` currently treats `tasks.owner` as a hard gate for executor delivery. That breaks when a task has been reassigned and the live executor role row is authoritative but `tasks.owner` is stale or reflects a prior handoff.

The visible failure mode is:

1. A reassigned executor tries to push their work.
2. `coord_commit_push` rejects because `tasks.owner != caller`.
3. No `commit_pushed` event is published.
4. `coord_ship_to_dev` later fails because it cannot find an executor commit for the task.

## Intended behavior

Executor delivery should be validated against the live executor role assignment for the task, not against stale task metadata.

The canonical source of truth for who may push is:

- task is in `execute`
- task belongs to the active project
- caller has an active executor role row on that task
- the active executor role row is uncompleted and unsuperseded

`tasks.owner` may remain as a historical/sticky value, but it should not block a valid executor push when the live role assignment belongs to the caller.

## Scope

### `coord_commit_push`

- Keep the stage and project validation.
- Change the ownership gate so the caller is authorized by the active executor role row.
- Preserve the existing wrong-caller rejection when the caller does not match the live executor assignment.
- Preserve the push-failure behavior: failed push still suppresses the task id in the published event so kanban does not auto-advance.
- Keep the normal success path publishing `commit_pushed` with the task id and completing the executor role row.

### `coord_ship_to_dev`

- No independent semantic change is expected.
- The ship-stage failure in this bug is downstream of the missing `commit_pushed` event.
- Add a regression that demonstrates ship succeeds once the reassigned executor can push successfully.

## Acceptance tests

Add or update tests to cover:

1. A task in `execute` whose `tasks.owner` is stale, but whose active executor role row belongs to the caller, is accepted by `coord_commit_push`.
2. The push publishes `commit_pushed` with the task id and completes the executor role row.
3. The wrong-caller case still rejects when the caller does not match the active executor role owner.
4. A follow-on ship path still succeeds after the reassigned executor push, proving the downstream `coord_ship_to_dev` failure was caused by the push gate.

## Non-goals

- Do not change role reassignment mechanics.
- Do not change shipper authorization rules unless a test proves they depend on stale task ownership.
- Do not rewrite task ownership history purely for this bug.
